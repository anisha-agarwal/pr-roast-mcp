# Code Review: "Fix keep-alive workflow"

**Severity: 🔥🔥**

---

## The Good News
You identified a real problem and fixed it. The workflow was actually broken. That's... something.

## The Bad News

**1. You're pinging a business logic function as a health check.**
This isn't a "safe no-op ping" — you're calling `get_family_by_invite_code` with a fake code. What if someone later adds logging, metrics, or audit trails to that RPC? Now your keep-alive job is polluting production data. You just turned a database function into a janky health endpoint. 🤦

**2. "Returns 200 for any input" is not a guarantee.**
You tested it locally once with one input. That's cute. What if the function gets updated? What if there's a constraint violation we don't know about? Production keep-alives silently fail and nobody notices until the project spins down. Did you add monitoring for this?

**3. Zero tests, zero documentation.**
You verified this locally with a curl. That's test plan? The only barrier between this breaking in 6 months is your memory. Add a comment explaining *why* this specific RPC is acceptable, or better yet, add it to the test suite.

**4. The real fix should've been different.**
Supabase changed their API — the right move is either:
- Use the service role key (you have it, right?)
- Create an actual health check endpoint
- Use `/rest/v1/health` or whatever Supabase provides
- Ask why the anon key even needs to keep the project alive

Instead you're reverse-engineering workarounds.

---

## The Compliment
At least you documented *what* changed and *why*. The comments are clear. That comment is doing heavy lifting to explain a terrible decision, but it's there.

**Ask before merging:** Why can't we use the service role key? And why is this the preferred solution over literally any other approach?
