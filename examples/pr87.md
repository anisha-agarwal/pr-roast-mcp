# Code Review: Streak Milestone Push Notifications

## 🔥🔥🔥 SEVERITY: Medium-High

---

## The Good Stuff (I'm not a monster)

✅ **Test coverage is thorough** — 184 lines of tests covering happy path, edge cases (null parents, no subscriptions), and error states. That's genuinely solid.

✅ **Error handling is thoughtful** — validates `days`, checks for family existence, handles `Promise.allSettled` for parent notifications gracefully.

---

## The... Concerns

### 1. **Fire-and-forget notification (🔥🔥)**
```typescript
void fetch('/api/push/trigger/streak-milestone', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ days, badge: result.badge }),
})
```
You're literally calling `void` on a fetch with zero error handling. If this fails, the user unlocks a badge, gets a toast, but no one gets notified—and you'll never know. Add `.catch()` at minimum. This is a "hope and pray" pattern.

### 2. **Parent payload is identical to child (🔥)**
Both child and parents get:
```typescript
body: `${profile.display_name} hit a ${body.days}-day streak`
```
Parents should get *different* messaging. "Your child Leo hit a 7-day streak!" not "Leo hit a 7-day streak" (which sounds like the parent is bragging about themselves). This is a UX miss.

### 3. **Test mock chain is... ambitious (🔥)**
```typescript
function selectChain(data: unknown) {
  const chain: Record<string, unknown> = {}
  chain.select = () => chain
  chain.eq = () => chain
  chain.single = () => Promise.resolve({ data, error: null })
  return chain
}
```
This works but is fragile as wet tissue. If someone adds a `.filter()` or reorders the query, it breaks silently. Consider using `jest-mock-extended` or similar. Also, `selectChain` vs `parentListChain` having different signatures is asking for bugs.

### 4. **Missing auth context (🔥)**
The endpoint validates `auth.getUser()` but doesn't verify the user is actually a child, not a parent manually triggering this. Could a parent call this for their kid? Should they not be able to?

---

## Minor Gripes

- `parentSent` counts *fulfilled promises > 0*, not actual parents notified. If a parent has 5 devices, they still count as 1. Is that intentional?
- No logging/observability beyond the `withObservability` wrapper — hard to debug failures in production
- `body?.badge ?? \`${body.days}-day streak\`` — using `??` after destructuring `body.badge` is redundant (just use `||`)

---

## Summary

**Ship it, but fix the fire-and-forget fetch and differentiate parent messaging.** The tests are great, but production will bite you if notifications silently fail.
